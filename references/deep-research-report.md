# LangChain, LangGraph, and Deep Agents

## From Chains to Harnesses

The cleanest way to understand this trio is as a historical and architectural stack. LangChain began in October 2022 as a Python package with two core ideas: model abstractions and **“chains,”** meaning predetermined sequences of LLM-related computation such as retrieval followed by generation. The project’s own philosophy page explicitly says the name “LangChain” comes from “Language” and “Chains.” By December 2022, LangChain had already added general-purpose agents inspired by ReAct; by March 2023 it had adapted to provider-native function calling; and by February 2024 LangGraph was released to provide the low-level orchestration layer LangChain lacked for exact control over long-running agent flows. In LangChain v1, the namespace was then streamlined around agent building blocks, while legacy functionality moved to `langchain-classic`. This is not just product churn; it mirrors how LLM software itself evolved: from fixed pipelines, to tool-using loops, to stateful runtimes, to batteries-included autonomous harnesses. citeturn5view0turn14view6

LangChain’s own conceptual documentation now formalizes the distinction. It frames **LangChain** as an agent framework, **LangGraph** as an agent runtime, and **Deep Agents** as an agent harness. In that framing, a framework gives abstractions and integrations, a runtime gives durable execution and persistence, and a harness adds opinionated prompts, predefined tools, context management, and subagents for more autonomous work. That official split is the right mental model for the rest of this report. citeturn3view0turn1view6turn2search2

## What LangChain Contributes

LangChain’s current center of gravity is the **agent harness** exposed by `create_agent`. Its docs define an agent as “a model calling tools in a loop until a given task is complete,” and then make the key conceptual move: **Agent = Model + Harness**. The harness is everything around the loop—the model, prompt, tools, and middleware—and LangChain’s standard entry point is intentionally minimal: configure `model=`, `tools=`, `system_prompt=`, and then extend behavior with middleware when needed. In v1, `create_agent` is presented as the standard way to build agents in LangChain, and as a simpler interface than lower-level LangGraph prebuilt agents while remaining highly extensible. citeturn14view4turn14view6

The important technical contribution of LangChain is therefore not “chains” in the old narrow sense, but a **common abstraction boundary** for models, tools, messages, structured output, and middleware. The tools docs describe tools as callable functions with explicit schemas and descriptions that are passed to the chat model; the model then decides when to invoke them and with what arguments. The context-engineering docs go further and make a strong claim: most agent failures are not caused by raw model incapability, but by giving the model the wrong context. That is why middleware is central in LangChain v1: it lets the developer control dynamic prompts, summarization, selective tool access, guardrails, and state shaping as composable concerns rather than burying them in ad hoc prompt code. citeturn5view3turn14view5turn14view3

Just as important, LangChain agents are **not** a separate runtime. The release notes emphasize that `create_agent` is built on LangGraph, so persistence, streaming, human-in-the-loop, and even time-travel-style replay come “for free” at the LangChain layer. In other words, LangChain is the ergonomic framework layer over a graph runtime. That design decision is exactly what makes Deep Agents possible later: a higher-level harness can be assembled on top of the same primitives without inventing a new execution substrate. citeturn14view6turn1view6turn9search21

## What LangGraph Contributes

LangGraph is the layer where agent execution stops looking like a chat loop and starts looking like a **program**. Its Graph API defines three fundamental pieces: **state**, **nodes**, and **edges**. State is the current snapshot of the application, nodes are functions that consume and update that state, and edges determine what node executes next. The docs are explicit that nodes and edges are “nothing more than functions,” which means a node can contain an LLM call, a tool invocation, or ordinary deterministic Python code. Under the hood, LangGraph models execution with message passing and discrete **super-steps**, inspired by Google’s Pregel model. Nodes become active when they receive updates, run, emit updates, and then halt until reactivated. This is the conceptual jump from chains to graphs: the control flow is no longer “A then B then C,” but an evolving state machine with loops, branches, and parallelism. citeturn14view1

Its second decisive contribution is **persistence**. LangGraph saves state as checkpoints. When a graph is compiled with a checkpointer, a snapshot of graph state is saved at every super-step, organized into threads keyed by `thread_id`. The persistence docs tie this directly to the features people want in production agents: human-in-the-loop approval, conversational memory, time-travel debugging, fault tolerance, and pending-write recovery when only some nodes in a parallel super-step succeed. The durable execution docs add an important technical constraint: if you want replay-safe resumption, non-deterministic operations and side effects need to be wrapped in tasks or otherwise made idempotent, because resumption is implemented by replaying from a valid restart point rather than literally continuing at the same line of code. citeturn16view0turn16view1

Its third contribution is **programmable interruption and branching**. Interrupts let a graph pause at any point, surface a JSON-serializable payload, and wait indefinitely until resumed with a `Command`. Time travel lets you replay or fork from a checkpoint, re-executing downstream work while keeping prior checkpoints intact. Subgraphs let one graph become a node in another graph, with explicit choices about whether the subgraph should be per-invocation, per-thread stateful, or fully stateless. Those features matter enormously for agents because they turn “agent behavior” into inspectable, resumable, branchable execution rather than a single opaque model call. Deep Agents inherits all of this because it runs on LangGraph. citeturn16view2turn16view3turn17view0

## What Deep Agents Adds

Deep Agents is LangChain’s answer to a very specific class of problem: **complex, long-horizon, open-ended work** such as research, coding, and multi-step task execution. The official overview describes `deepagents` as a standalone library built on LangChain’s core agent building blocks and using the LangGraph runtime for durable execution, streaming, and human-in-the-loop. The product page and overview both describe it as an **agent harness** with built-in planning, filesystem-backed context management, subagent spawning, and long-term memory. The key phrase from the docs is that it is the *same core tool-calling loop* as other agent frameworks, but with built-in capabilities that make agents more reliable on real tasks. citeturn1view0turn7search2turn23view4

The API surface makes that concrete. `create_deep_agent` accepts a model, tools, `system_prompt`, middleware, subagents, skills, memory, permissions, a backend, `interrupt_on`, `response_format`, `context_schema`, a checkpointer, a store, debug flags, name, and cache, and returns a **`CompiledStateGraph`**. That return type is a subtle but important fact: Deep Agents is not “notebook glue” around a chat model. It is a graph-compiled harness that still exposes LangGraph runtime concepts such as checkpointing and stores to the caller. In practice, Deep Agents is opinionated at the harness layer but honest about the runtime layer beneath it. citeturn25view0turn11search0

The harness docs group its built-ins into four buckets: **execution environment**, **context management**, **delegation**, and **steering**. Execution environment means tools, virtual filesystem, optional sandbox, and interpreter. Context management means skills, memory, summarization, context offloading, and prompt caching. Delegation means planning plus subagents. Steering means human-in-the-loop approval and interrupts. That grouping is not marketing fluff; it is the real architecture. Deep Agents’ core insight is that long-running agents fail mostly because the *environment around the loop* is underdesigned—especially context discipline—not because the loop itself is mysterious. citeturn3view1turn21view1turn23view4

Deep Agents is also deliberately **model-neutral**. The models docs say Deep Agents work with any LangChain chat model that supports tool calling, and maintain a Deep Agents eval suite for basic agent operations. At the same time, the docs warn that passing those evals is necessary but not sufficient for strong long-horizon performance. That is an important design signal: the harness assumes model interchangeability at the API boundary, but it also openly acknowledges that quality depends on more than simple tool-call correctness. citeturn18view2turn7search17

## How Deep Agents Is Assembled

The clearest way to see Deep Agents’ internals is as a **middleware stack over LangChain’s `create_agent`**. The customization docs say Deep Agents supports arbitrary middleware, but by default gives the agent a specific set: `TodoListMiddleware`, `FilesystemMiddleware`, `SubAgentMiddleware`, `SummarizationMiddleware`, `AnthropicPromptCachingMiddleware`, and `PatchToolCallsMiddleware`. This list captures the entire thesis of the harness: explicit planning, externalized context, task delegation, automatic context compression, provider-specific cost/latency optimization, and repair of interrupted or cancelled tool-call histories. citeturn27view0

Planning is more precise than many people realize. The `TodoListMiddleware` lives in the LangChain codebase and adds a `write_todos` tool for structured task lists. Its code says it injects its own system prompt instructions and enforces that `write_todos` can be called **at most once per model turn**, because the tool replaces the entire todo list and parallel updates would make precedence ambiguous. So the planning layer is not just “agent makes a plan”; it is a stateful, tool-mediated planning channel with an update discipline. citeturn25view1

The filesystem layer is equally opinionated. The `FilesystemMiddleware` adds `ls`, `read_file`, `write_file`, `edit_file`, `glob`, and `grep`, and adds `execute` if the backend supports sandbox execution. The middleware code says it automatically evicts large tool results to the filesystem when they exceed a threshold, replacing them with truncated previews and file references. In the current implementation, the constructor defaults are **20,000 tokens** for tool-result eviction, **50,000 tokens** for large human-message eviction, and a maximum `execute` timeout of **3,600 seconds**. The code also defines internal roots such as `/large_tool_results` and `/conversation_history`. This is one of the deepest ideas in the harness: context is treated as something the system can *externalize and rehydrate*, not just something the model passively consumes. citeturn12view1turn20view4turn20view3

That same idea appears in the conceptual context-engineering docs. Deep Agents distinguishes **input context** from **runtime context** from **context compression**. Input context is what goes into the startup system prompt: your custom instructions, memory, skills, and tool prompts. Runtime context is per-run configuration that propagates through the runtime and to subagents, but is not automatically shown to the model unless tools or middleware decide to surface it. Compression then happens through two built-in strategies: **offloading** and **summarization**. Offloading uses filesystem references to replace huge tool inputs/results once the context nears the model limit; summarization kicks in when the context crosses roughly **85%** of the model profile’s `max_input_tokens`, keeps about **10%** as recent context, and can also trigger reactively after a context overflow error. The docs also expose an optional summarization tool middleware so the agent can summarize at semantically good moments instead of only at fixed thresholds. citeturn19view3turn21view0turn19view5

Prompt engineering in Deep Agents is therefore not just “write one big system prompt.” The customization docs explicitly warn against copying the built-in prompt verbatim because the real value of Deep Agents comes from the orchestration layer the SDK provides. Your `system_prompt` is **prepended** to the built-in prompt, and the final prompt is assembled from named parts so that caller instructions, SDK defaults, and model-specific profile overrides have stable precedence. In practice, most users see a `USER` slot, a `BASE` slot, and possibly a model-specific `SUFFIX`; harness profiles can also replace the base prompt outright. This named-slot assembly is one of the reasons the harness can stay opinionated while still being customizable. citeturn27view1turn26view1turn26view2turn21view5

Memory and skills are intentionally different concepts. Deep Agents’ long-term memory is **filesystem-backed**, not implicitly vector-memory-backed: the docs describe memory as files—often `AGENTS.md`—that the agent reads and writes through backends. Memory is always loaded when configured and is meant for persistent instructions, preferences, and shared state across conversations. Skills, by contrast, are directories with `SKILL.md` files plus optional scripts, docs, templates, or other assets. At startup the agent reads skill frontmatter only, then loads the full skill content only when it decides the skill is relevant. The docs call this **progressive disclosure**, and it is central to the harness’s approach to context economy. citeturn27view4turn27view6turn19view4

The backend system is the file- and memory-routing plane that makes those abstractions real. By default, the filesystem is thread-scoped in LangGraph state, which means files persist within a checkpointed thread but not across threads. `FilesystemBackend` exposes a real disk root, `StoreBackend` maps paths into a LangGraph store for cross-thread persistence, and `CompositeBackend` routes different path prefixes to different backends. The docs repeatedly recommend routing persistent memory under `/memories/` while keeping ephemeral working files elsewhere, and they note that `FilesystemBackend` by itself will mix internal agent artifacts such as offloaded tool results and conversation history into your project directory unless you isolate it with `CompositeBackend`. Namespace factories on `StoreBackend` let you scope data by assistant, by user, or by thread. Newer versions prefer backend instances directly; callable backend factories are deprecated and scheduled for removal in Deep Agents `0.7`. citeturn27view5turn29view1turn29view3turn29view5

Delegation is where Deep Agents stops being merely a nicer `create_agent` wrapper and becomes a genuine harness. The `SubAgentMiddleware` code adds a `task` tool and injects a subagent system prompt that teaches the main agent when to delegate, when not to, and to **parallelize independent work whenever possible**. The subagents docs frame the reason clearly: context quarantine. When a subagent does heavy research or large-file analysis, the main agent gets back a clean result rather than all the intermediate tool chatter. Dictionary-defined subagents specify at least a name, description, system prompt, tools, and optionally a different model; profiles can also edit or disable the built-in general-purpose subagent that Deep Agents can auto-provision. citeturn12view0turn21view2turn21view3

Async subagents extend that idea into distributed, non-blocking work. They are a preview feature introduced in `deepagents` 0.5, and the docs define them as background tasks launched over the Agent Protocol, either in-process over ASGI or remotely over HTTP. The supervisor gets five tools—`start_async_task`, `check_async_task`, `update_async_task`, `cancel_async_task`, and `list_async_tasks`—and the middleware handles thread creation, run management, and persistence. A particularly elegant implementation detail is that async task metadata lives in a dedicated state channel, `async_tasks`, rather than only in message history, specifically so task IDs survive message-history compaction and summarization. That is classic Deep Agents design: if something is operationally important, do **not** leave it in plain chat history and hope the model remembers it. citeturn21view4turn30view4turn30view6

Steering and safety are layered rather than magical. Human-in-the-loop uses LangGraph interrupts under the `interrupt_on` parameter and requires a checkpointer so execution can pause and resume on the same thread. Permissions let you declare allow/deny rules over built-in filesystem tools, with rules evaluated in order and the first match winning; notably, the docs warn that permissions do **not** cover custom tools, MCP tools, or sandbox execution. For stronger isolation, sandboxes lift the agent into an isolated execution environment and add `execute`; interpreters provide an in-process QuickJS-based `eval` tool for lightweight, in-loop programming and tool composition without giving the agent a full shell. The repository README adds a blunt but wise security principle: Deep Agents follows a **“trust the LLM”** model, so boundaries must be enforced at the tool and sandbox level rather than by expecting the model to self-police. citeturn21view6turn8view2turn21view7turn21view8turn6search5

Finally, Deep Agents also supports **structured output**. The customization docs say that if you pass a schema as `response_format`, the harness validates the output and stores it in the graph state under `structured_response`. That matters because it shows the harness is not only about autonomy; it can still terminate in typed, application-friendly outputs. citeturn1view2turn25view0

## How Deep Agents Has Evolved

The harness’s evolution over the last several releases is revealing. In the October 2025 `0.2` release, LangChain explicitly positioned Deep Agents above LangChain and LangGraph and highlighted **pluggable backend abstractions** plus **composite backends** as the major new capability. That was an early signal that the team saw long-horizon agent reliability as a context-management and storage problem, not only a prompt problem. citeturn23view0

In February 2026, `deepagents` `0.4` added **pluggable sandbox support**, **smarter conversation summarization**, and made the **OpenAI Responses API** the default for OpenAI models. The changelog also described two sandbox integration patterns—running the agent inside the sandbox versus running the agent externally and using the sandbox as a remote tool—and recommended the latter for cleaner credential boundaries and operational flexibility. That release sharpened the division between the control plane of the harness and the execution plane of the environment. citeturn23view1

In April 2026, `0.5` added **async subagents**, described as non-blocking remote delegation for tasks that take minutes rather than seconds. Around the same time, LangChain introduced **harness profiles**, with the company blog saying that model-specific prompt/tool/middleware tuning produced a **10–20 point** gain on a subset of `tau2-bench` over the previously generic harness. The same post says Deep Agents ships profiles for OpenAI, Anthropic, and Google models and that prompting guides and harness conventions differ enough by model family to move benchmark outcomes materially. This is a crucial development detail: Deep Agents has matured from a static batteries-included harness into a **model-conditioned harness**. citeturn8view6turn23view2turn21view5

The current docs also recommend **event streaming** for new applications, describing it as the typed-projection API introduced in Deep Agents `v0.6`, where subagents, messages, tool calls, and values can be consumed through separate iterators. GitHub releases showed `deepagents==0.6.4` as the latest release on May 26, 2026, and the platform docs now also describe **Managed Deep Agents** in private preview as an API-first hosted runtime in LangSmith. That managed path keeps the familiar project shape—`AGENTS.md`, `skills/`, `subagents/`, and `tools.json`—but adds the operational layer around the open-source harness. citeturn21view9turn6search2turn23view5

## How to Build Great Systems with It

The first design rule is to choose the right layer. Use **LangChain** when you want a lightweight harness and team-standard abstractions. Use **LangGraph** when the loop itself is not the right shape and you need explicit control over state transitions, parallelism, resumption, or bespoke workflow logic. Use **Deep Agents** when the task is open-ended, long-running, and primarily bottlenecked by context discipline, decomposition, memory, or execution environment rather than by the absence of a custom graph. LangChain’s own product comparison says essentially exactly that. citeturn3view0turn23view4

The second rule is to treat **memory as a filesystem routing problem**. Keep short-term working context in thread checkpoints; route long-term memory into `/memories/` through a `StoreBackend` or another durable backend; and decide intentionally whether memory should be assistant-scoped, user-scoped, or organization-scoped. The memory docs emphasize that memory writes can happen on the hot path during conversation or through a background consolidation agent on a cron schedule. They also warn that concurrent writes are effectively last-write-wins, so agent-scoped or org-scoped memory benefits from consolidation or topic-level file partitioning. This is very different from the casual “just add memory” advice that floats around agent discourse. citeturn8view0turn27view4turn28view0turn28view1turn28view3turn28view4turn28view5

The third rule is to keep **memory small and skills specific**. The docs say memory is always injected, while skills are progressively disclosed; that means memory should hold stable, always-relevant conventions and identity, while skills should package reusable workflows, domain knowledge, and auxiliary assets that only need to load when relevant. If you put everything into memory, you are fighting the harness; if you split every tiny preference into a skill, you are also fighting the harness. Deep Agents is optimized for a narrow always-on memory layer plus a larger on-demand skill layer. citeturn19view3turn27view6turn27view4

The fourth rule is to treat **subagents as context quarantine, not fashion**. The docs explicitly say not every complex task needs multi-agent design. Sync subagents are best when the supervisor truly needs the result before continuing; async subagents are best when the user should not block and the work can continue independently. Either way, the discipline is the same: delegate work whose intermediate steps do not deserve to remain in the main context. If intermediate state is something the main agent must keep reasoning over step by step, a custom LangGraph workflow may be the better abstraction. citeturn21view2turn30view4turn17view0

The fifth rule is to build safety as **environmental policy**, not prompt aspiration. Deep Agents gives you path-based permissions for built-in filesystem tools, interrupt-based approvals for risky operations, backend policy hooks for custom enterprise rules, and sandboxes for true execution isolation. But the docs and repo are quite direct that these layers are complementary and that custom tools or shell execution require their own controls. In practice, the safest Deep Agents systems use `CompositeBackend` to separate artifacts from project files, permissions for built-in filesystem tools, `interrupt_on` for irreversible actions, and a sandbox whenever arbitrary code or shell execution is in play. citeturn8view2turn29view0turn21view6turn21view7turn29view1

The sixth rule is to instrument the system as a **runtime**, not as a chatbot. Because Deep Agents compiles to LangGraph, you can stream values, tool calls, messages, and subagent updates, and the newer event streaming API exposes typed projections for each. That matters for observability and UI design: long-running agents should be rendered as progressing stateful processes, not just as a token stream. This is exactly the design that Deep Agents Code demonstrates in a packaged form: the `dcode` CLI is a coding agent built on the SDK, with provider-agnostic models, persistent memory, approvals, skills, and optional web search. It is useful to think of `dcode` as proof that the harness is not just a research abstraction; it is meant to be productized. citeturn21view9turn18view7turn1view9turn7search8

## Open Questions and Limits

A few pieces of the current stack are intentionally still in motion. **Async subagents** are documented as a preview feature whose APIs may change. **Interpreters** are marked experimental. **Managed Deep Agents** is in private preview. And Deep Agents’ permissions model does **not** cover custom tools, MCP tools, or sandbox execution, which means serious production deployments still need explicit backend and environment policy design. citeturn21view4turn21view8turn23view5turn8view2

There are also some practical maturity signals to keep in mind. The current docs say suggested models are drawn from a Deep Agents eval suite, but passing those evals is not sufficient for strong performance on longer tasks. The backends docs also note an ongoing migration away from callable backend factories toward direct backend instances, with the old pattern deprecated ahead of `0.7`. So while the harness architecture is already coherent, some of its operational APIs are still tightening. citeturn18view2turn29view5

The most important conclusion, though, is stable: **LangChain, LangGraph, and Deep Agents are not three unrelated products. They are three layers of the same answer to increasingly difficult LLM software problems.** LangChain standardizes the agent loop and surrounding abstractions. LangGraph makes that loop durable, stateful, interruptible, and inspectable. Deep Agents then codifies the hard-won lessons of long-horizon autonomy—planning, context compression, virtual filesystems, memory, subagents, sandboxes, and steering—into a reusable harness. If you keep that stack in your head, the library ecosystem stops feeling sprawling and starts feeling almost inevitable. citeturn3view0turn1view0turn14view6turn14view0turn23view4
